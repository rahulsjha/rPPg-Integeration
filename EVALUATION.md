# Accuracy and Validation Framework

This document defines exactly how accuracy is measured for the near real-time rPPG pipeline and how to reproduce the results.

## 1) Why Accuracy Needs a Formal Protocol

A single BPM number is not enough to evaluate an rPPG system. A production-style evaluation must report:

- chunk-level errors over time,
- final aggregated error,
- robustness indicators,
- runtime constraints.

This project supports all four.

## 2) Evaluation Inputs

Required:

- Prediction JSON generated by [run_chunked_prototype.py](run_chunked_prototype.py)

Optional for true accuracy:

- Ground-truth heart-rate CSV with columns: time_sec,bpm
- Ground-truth respiratory rate scalar (breaths/min)

## 3) Metrics Definitions

### 3.1 Chunk-Level Error Metrics

Given chunk predictions \(\hat{y}_i\) and reference chunk targets \(y_i\):

- Mean Absolute Error (MAE)

$$
\mathrm{MAE} = \frac{1}{N}\sum_{i=1}^{N}|\hat{y}_i - y_i|
$$

- Root Mean Squared Error (RMSE)

$$
\mathrm{RMSE} = \sqrt{\frac{1}{N}\sum_{i=1}^{N}(\hat{y}_i - y_i)^2}
$$

- Mean Absolute Percentage Error (MAPE)

$$
\mathrm{MAPE}(\%) = \frac{100}{N}\sum_{i=1}^{N}\left|\frac{\hat{y}_i-y_i}{y_i}\right|
$$

- Threshold Accuracy

$$
\mathrm{Acc}_{\pm k}(\%) = \frac{100}{N}\sum_{i=1}^{N}\mathbf{1}\left(|\hat{y}_i-y_i| \le k\right)
$$

where \(k\in\{3,5,10\}\) bpm.

- Pearson Correlation (temporal tracking quality)

$$
r = \mathrm{corr}(\hat{y}, y)
$$

### 3.2 Overall BPM Error

Overall prediction error against session-level reference:

$$
\mathrm{OverallAbsError} = |\hat{y}_{\mathrm{overall}} - y_{\mathrm{overall}}|
$$

### 3.3 Respiratory Rate Error (if reference available)

$$
\mathrm{RRAbsError} = |\hat{r} - r|
$$

### 3.4 No-Ground-Truth Proxy Quality

If reference data is unavailable, we still report stability and confidence proxies:

- median confidence
- chunk BPM standard deviation
- chunk BPM interquartile range (IQR)
- outlier chunk ratio

These are quality indicators, not true accuracy.

## 4) Runtime Metrics Definitions

From prediction output:

- avg_chunk_compute_ms: mean chunk processing latency
- p95_chunk_compute_ms: 95th percentile chunk latency
- effective_pipeline_fps: processed frames per wall-clock second
- realtime_factor_x:

$$
\mathrm{RTF} = \frac{\text{video duration (s)}}{\text{wall time (s)}}
$$

\(\mathrm{RTF} \ge 1\) indicates at-least-real-time throughput.

## 5) Recommended Acceptance Targets

For a practical prototype target:

- MAE <= 8 bpm
- RMSE <= 12 bpm
- Within ±10 bpm >= 80%
- Overall BPM abs error <= 7 bpm
- RTF >= 1.0 for true real-time deployment goal

These targets may be tightened for controlled datasets.

## 6) Reproducible Commands

### 6.1 Generate Prediction Output

```bash
python3 run_chunked_prototype.py \
  --video input/assignment_60s_exact.mp4 \
  --chunk-sec 5 \
  --json-out notes/chunked_rppg_output.json
```

### 6.2 Evaluate Without Ground Truth (Proxy Mode)

```bash
python3 evaluate_accuracy.py \
  --pred notes/chunked_rppg_output.json \
  --out notes/accuracy_report.json
```

### 6.3 Prepare Ground Truth and Evaluate (True Accuracy)

```bash
cp notes/reference_hr_template.csv notes/reference_hr.csv
# Edit notes/reference_hr.csv to match your real reference stream
python3 evaluate_accuracy.py \
  --pred notes/chunked_rppg_output.json \
  --ref-bpm-csv notes/reference_hr.csv \
  --out notes/accuracy_report.json
```

### 6.4 Evaluate With Ground-Truth BPM + RR

```bash
python3 evaluate_accuracy.py \
  --pred notes/chunked_rppg_output.json \
  --ref-bpm-csv notes/reference_hr.csv \
  --ref-rr 14.8 \
  --out notes/accuracy_report.json
```

## 7) Ground Truth CSV Format

Use [notes/reference_hr_template.csv](notes/reference_hr_template.csv) as the starter template.

File example for [notes/reference_hr.csv](notes/reference_hr.csv):

```csv
time_sec,bpm
0.0,78
1.0,79
2.0,80
...
```

The evaluator averages reference BPM inside each predicted chunk window.

## 8) Interpretation Guidance

- Low MAE and high within-5/within-10 indicate strong practical quality.
- High correlation with moderate MAE indicates good trend tracking but calibration offset.
- High outlier ratio often indicates motion/lighting issues or unstable ROI.
- RR should be interpreted as lower confidence than BPM in RGB-only pipelines.

## 9) Common Failure Modes

- Subject motion, talking, yaw/pitch changes
- Nonuniform illumination and flicker
- ROI drift with weak face detection
- Compression artifacts in low-quality videos

## 10) Reporting Template

For submissions, include:

- chunk-level BPM table,
- overall BPM,
- MAE/RMSE/MAPE/within-threshold/correlation,
- runtime metrics and RTF,
- failure-case notes and mitigation plan.
